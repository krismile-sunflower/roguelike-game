# 本仓库 GDScript 约定

这份文档不是通用 GDScript 教材，而是当前项目的代码约定说明。目标是让接手开发的人快速理解这个仓库里“怎么写、怎么组织、哪里容易出问题”。

## 书写风格

当前项目默认采用 typed GDScript。

建议保持：

- 明确的变量类型标注
- `@onready` 缓存场景节点
- 常量集中放在文件顶部
- 信号名称语义清晰
- 以小函数拆分流程，而不是把所有逻辑塞进 `_ready()` 或 `_input()`

示例风格：

```gdscript
const TEXT_START := "开始整理"

@onready var start_button: Button = $MenuLayer/MenuPanel/StartButton

var current_level_index: int = 0
```

## 当前常见模式

### 1. AutoLoad 单例访问

当前全局状态与服务通过 AutoLoad 访问：

- `GameData`
- `GameState`
- `SaveManager`
- `AudioManager`
- `ParticleManager`

使用原则：

- 全局进度和状态通过单例存
- 场景内局部表现不要滥用单例
- 只有明确跨场景共享的数据再进入 AutoLoad

### 2. `CanvasLayer + Control` UI 分层

当前主界面把菜单、说明、暂停、教程、HUD、完成层都放在 `main.tscn` 内部。

这意味着：

- UI 结构集中，适合原型期快速迭代
- 输入层容易互相影响
- `mouse_filter` 是高频敏感点

### 3. 拖放交互集中处理

当前拖放并不是每个物品自己接管输入，而是由 `main.gd` 统一处理鼠标事件：

- 找出被点击的物品
- 标记为当前拖拽物
- 随鼠标移动
- 松手后进行目标判定

这个模式的优点是：

- 全局拖放规则更容易统一
- 更适合小规模整理玩法

代价是：

- 输入、命中、判定都集中在主场景脚本里
- 主脚本变大后要注意继续拆分

### 4. Tween 作为轻反馈主力

当前项目用 Tween 处理：

- 吸附回位
- 提示闪动
- 完成面板弹出

原型阶段优先保持轻量、可读、可快速调试，不急着引入更重的动画系统。

## 当前易踩坑

### `_input()` 与 `_unhandled_input()`

整理玩法的拖拽对输入链路很敏感。

当前经验是：

- 需要稳定接收拖拽事件时，优先使用 `_input()`
- `Control` 多层覆盖时，`_unhandled_input()` 很容易拿不到鼠标事件

### `mouse_filter`

这是当前仓库最容易导致“物品拖不动”的问题来源之一。

重点检查：

- 说明层遮罩
- 教学层
- HUD 面板
- 完成层遮罩
- 非交互文案节点

原则是：

- 只有真的需要点击的控件才拦截输入
- 装饰控件和说明文字尽量忽略鼠标

### 字典取值的类型推断

当前关卡数据大量使用字典。

当 `Dictionary.get()` 返回值不够明确时，Godot 可能无法自动推断类型。遇到这类情况，优先显式声明变量类型，例如：

```gdscript
var paper_color: Color = Color(level_data.get("background_color", Color.WHITE))
```

不要过度依赖隐式推断，尤其是在颜色、向量、数组这类地方。

### 中文文案编码稳定性

当前项目已经转向中文界面，文案维护要避免再次出现乱码污染。

建议：

- Markdown 文档全部保存为 UTF-8
- 脚本中的玩家可见文案尽量集中定义
- 修改中文文案后，优先在编辑器里实际看一遍显示效果

## 命名与组织约定

- 脚本文件名：英文，小写加下划线
- 场景文件名：英文，小写加下划线
- 内部 `id`：英文稳定值
- 玩家可见文本：中文
- 一个主场景配一个主控制脚本

当前典型对应关系：

- `main.tscn` ↔ `main.gd`
- `draggable_item.tscn` ↔ `draggable_item.gd`
- `drop_target.tscn` ↔ `drop_target.gd`

## 何时改代码，何时改数据

当前原型里，很多内容仍写在 `LEVELS` 常量中。

建议边界如下：

- 改关卡标题、目标、物品摆放：优先改数据
- 改拖放规则、状态流、菜单行为：改脚本逻辑
- 改颜色和基础反馈：通常先改脚本里的常量或样式构造

## 当前代码维护建议

- 尽量保持 `main.gd` 的流程函数清晰，不要继续堆叠匿名逻辑
- UI 文案、按钮连接、样式配置继续集中管理
- 对仍保留但不再使用的旧玩法脚本，除非明确要清理，否则不要重新接回主流程
- 修改拖放、菜单、暂停、教程时，记得一起检查存档和 HUD 是否跟上
